# ralphi — 점검

## Step 1: 모드 판별

발화에 "고쳐", "수정해", "fix", "처리해" 포함 → **자율** (수정까지 한다)
그 외 → **HITL** (보고만 하고 대기)

## Step 2: 스코프 판별

- 파일 경로가 주어짐 → **파일 스코프** → Step 3A
- 디렉토리가 주어짐 → **디렉토리 스코프** → Step 3B
- 아무것도 없음 → **사용자에게 묻는다**: *"파일 경로 또는 디렉토리 경로를 주세요. (`전체` / `다` 같은 응답은 유효 스코프가 아닙니다 — 구체 대상을 지정해주세요.)"* 추측 금지. git diff를 멋대로 돌리지 않는다.

## Step 3A: 파일 점검

1. 타입 감지 → criteria 파일 결정:
   - `docs/prd-*.md` (§섹션 구조 확인) → `criteria-prd.md`
   - `*/SKILL.md` (frontmatter name: 확인) → `criteria-skill.md`
   - `*.sh` (shebang #!/ 확인) → `criteria-script.md`
   - `.py .ts .js .go` 등 소스코드 → `criteria-code.md`
   - 불명확 → **사용자에게 묻는다**: *"감지된 파일 `{경로}` — 어떤 타입인가요? (prd / skill / script / code / 기타?)"* 추측 금지.
2. `references/` 에서 위 criteria **1개만** 로드. 다른 criteria 읽지 않는다.
3. 정적 분석 (구조, 포맷, 계약 위반)
4. 동적 실행 (스크립트/코드인 경우)
5. **필요성 검증 (소크라테스식)** — 아래 공통 메서드 참조.
6. → Step 4로

## Step 3B: 프로젝트/디렉토리 점검

1. 테스트 실행. 코드를 읽기 전에 테스트부터. 테스트 러너 모르면 **사용자에게 묻는다**: *"테스트 명령은? (예: `pytest`, `npm test`, `go test ./...`, `cargo test`, 커스텀?)"* 추측 금지. 도구 미설치(command not found) → 해당 검사 SKIP + 경고. **설치 시도 금지.**
2. 변경 파일 목록 (git diff)
3. 각 파일의 **diff만** 분석. **파일 전체를 읽지 않는다.**
4. 시나리오 워크스루 (의도 vs 구현). 해당 함수만 읽는다. 호출 그래프 추적 금지.
5. 커버리지 갭 탐색
6. **필요성 검증 (소크라테스식)** — 아래 공통 메서드 참조. 컴포넌트 단위 대신 diff hunk 단위로 적용.
7. → Step 4로

## 필요성 검증 (소크라테스식) — 공통 메서드

Step 3A.5(컴포넌트 단위)와 Step 3B.6(diff hunk 단위)에서 사용.

4가지 질문:

1. **왜** 여기 있지? — *(목적)*
2. **무엇으로** 구성돼 있지? — *(구성, Q1/Q4 답변을 위한 scaffold)*
3. 포장을 벗기면 **사실은** 뭐지? — *(진실, Q1/Q4 답변을 위한 scaffold)*
4. **없으면** 어떻게 되지? — *(필요성)*

**판정**: Q1과 Q4만 게이트다. Q2와 Q3는 정직한 답에 도달하기 위한 scaffold.
Q1 또는 Q4에서 답을 못 내면 → `[warning] 정당화되지 않은 존재` (관습/상속/권위로만 유지되는 항목).

## Step 4: 모드 분기

### HITL인 경우

Step 5 포맷으로 보고 → 사용자 판단 대기.

#### 사용자 응답 해석 규칙

명확한 지시 (즉시 실행):
- "1번 고쳐" → 해당 이슈만 수정 → 테스트 → 보고.
- "1, 3 고쳐" → 해당 이슈들만 순차 수정, 매번 테스트.
- "다 고쳐" → 전체 순차 수정, 매번 테스트.
- "무시해" / "넘어가" → 해당 이슈 기록 후 종료.

모호한 응답 (재질문 필수):
- "응", "알겠어", "그래", "오케이" → **수정 허가로 해석 금지.**
  → 재질문: "어떤 이슈를 수정할까요? 번호를 지정하거나 '다 고쳐'라고 해주세요."
- 이슈와 무관한 응답 → 재질문.

### 자율인 경우

critical→warning→coverage 순 즉시 수정 → 매 수정 후 테스트 → 전부 끝나면 Step 5 포맷으로 결과 보고.
- 테스트 FAIL → 해당 수정 롤백, 미수정 "테스트 실패"
- 의도 불명확 (코드 삭제, 로직 변경) → 미수정 "의도 불명확"
- 구조 변경 필요 (파일 이동, 인터페이스 변경) → 미수정 "구조 변경 필요"

**수정은 빼는 것을 우선한다.** `[정당화되지 않은 존재]` 항목의 기본 동작은 제거. `[critical/warning]` 위반에 대해서는 **최소한의 추가 패치**(null check, 누락 import, 타입 캐스트, 에러 가드)는 허용 — criteria를 만족시키는 데 필요한 경우. 단 새 추상화, 새 모듈, 새 기능 도입 금지. 새 추상화/모듈/기능이 필요한 수정은 `[미수정] 구조 변경 필요`로 표시.

미수정 건이 있으면 사용자 판단으로 넘긴다.

## Step 5: 보고

심각도: `critical`(동작 오류) / `warning`(잠재 문제) / `coverage`(테스트 갭)

HITL:
```
## ralphi 점검 결과
대상: {경로 | "사용자 지정 범위"}
### 발견 ({N}건)
1. [{심각도}] {파일:라인} — {원인 한 줄}
어떤 이슈부터 수정할까요?
```

자율:
```
## ralphi 점검 + 수정 결과
대상: {경로 | "사용자 지정 범위"}
### 수정 완료 ({M}/{N}건)
1. [fixed] {파일:라인} — {수정 내용 한 줄}
### 미수정 ({K}건) ← 있을 때만
1. [{심각도}] {파일:라인} — {원인} → {미수정 사유}
테스트: {PASS | FAIL (상세)}
```

이슈 없음 → `ralphi 점검 완료, 이상 없음.` 한 줄. 나열 금지.

## Context Budget

| 시점 | 읽는 것 |
|------|---------|
| Step 3A (파일 점검) | `references/`에서 정확히 **1개** criteria 파일, 감지된 타입에 매칭. 다른 파일 금지. |
| Step 3B (프로젝트 점검) | `git diff`만. **파일 전체 읽기 금지.** 시나리오 워크스루에 필요한 함수만. |
| 필요성 검증 | 이미 로드된 내용에 대해서만 동작. 새 파일 읽기 없음. |
| Step 5 (보고) | 파일 읽기 없음. 출력만. |

## 금지

- 변경 안 된 파일 읽기
- criteria 2개 이상 동시 로드
- HITL에서 지시 없이 수정
- 프로젝트 스코프에서 파일 전체 읽기 (diff만)
- "참고로..." 식 부가 정보
- 장황한 보고
