# ralpi — 아티팩트 정합성 검증

> "이 결과물이 올바른가?"를 끝까지 추적하는 자가검증 루프.
>
> 누가 만들었든 상관없다. PRD든, 코드든, 스킬이든 — ralpi는 검증한다.

## 1. 검증 대상과 위치

```
drafti → PRD 생성
           ↓
        [ralpi] → "이 PRD 올바른가?"
           ↓
        harnish → 자율 구현
           ↓
        [ralpi] → "구현이 올바른가?"
           ↓
        경험 축적 → 스킬 생성
           ↓
        [ralpi] → "이 스킬 올바른가?"
```

harnish, drafti와 **독립된 궤도**에서 동작한다. 직접 호출하지 않고, 사용자가 명시적으로 검증을 요청하면 실행한다.

## 2. 검증 루프 (핵심 흐름)

> **수정 주체**: ralpi가 직접 수정한다 (자가검증 + 자가수정 루프).
> 사용자 개입: 7회 초과 시에만 미해결 이슈를 보고하고 수동 수정을 요청한다.

```
[아티팩트 입력]
    ↓
① 타입 감지 (PRD | SKILL.md | 스크립트 | 구현 코드)
    ↓
② 타입별 검증 기준 로드 (references/criteria-*.md)
    ↓
┌→ ③ 정적 분석 (구조, 포맷, 모호 표현, 계약 위반)
│       ↓ 이슈 발견
│   ④ ralpi가 직접 수정
│       ↓
│   ⑤ 동적 실행 (스크립트/코드인 경우 실제 실행)
│       ↓ 이슈 발견
│   ⑥ ralpi가 직접 수정
│       ↓
│   PASS? ── YES → 종료 (PASS 리포트)
└──────── NO  (최대 7회, 초과 시 STOP + 미해결 이슈 목록 → 사용자에게 보고)
```

## 3. 타입 감지 규칙

| 입력 | 판별 기준 | 타입 |
|------|----------|------|
| `docs/prd-*.md` | 파일명 패턴 + §1~§8 섹션 구조 | **PRD** |
| `*/SKILL.md` | frontmatter `name:` + `description:` | **SKILL** |
| `*.sh` | shebang `#!/` + 실행 권한 | **스크립트** |
| 소스 코드 (`*.py`, `*.ts`, `*.tsx`, `*.js`, `*.java`, `*.kt`, `*.go`, `*.rs`, `*.swift`, `*.dart` 등) | 확장자 + 대응하는 테스트 파일 존재 여부 | **코드** |
| 그 외 | 사용자에게 타입 확인 요청 | — |

## 4. 타입별 검증 기준

### 4.1 PRD 검증

**정적 분석:**
- §4(구현 명세) + §6(테스트 기준) + §7(가드레일) 존재
- 섹션 번호 일관성 (순서 맞는가)
- 모호 표현 검사: "적절히", "필요하면", "나중에", "등등" → 구체화 요구
- §4 파일 경로가 실제 존재하거나 create로 표기되었는가
- §6 테스트 기준이 bash 실행 가능하거나 명확한 조건인가
- §7 가드레일에 hard/soft 구분이 있는가

**동적 실행:** 없음 (PRD는 문서)

상세: `references/criteria-prd.md`

### 4.2 SKILL.md 검증

**정적 분석:**
- frontmatter: `name`, `description` 필수 필드
- 모호 표현 0건
- if/then 분기가 100% 커버 (모든 조건에 대응 액션 존재)
- bash 경로가 실존하는가 (`${CLAUDE_SKILL_DIR}/scripts/*.sh`)
- 맥락 예산 섹션 존재
- references/ 파일이 실제 존재하는가

**동적 실행:**
- SKILL.md에 포함된 bash 명령이 실행 가능한가 (dry-run)
- 스크립트 경로 해석 정상 여부

상세: `references/criteria-skill.md`

### 4.3 스크립트 검증

**정적 분석:**
- `set -euo pipefail` 또는 동등한 에러 핸들링
- 인자 완전성 (필수 인자 누락 시 usage 출력)
- 출력 포맷이 소비자 계약에 맞는가
- shellcheck 통과 (설치돼 있으면)

**동적 실행:**
- `--help` 또는 인자 없이 실행 → usage 출력 확인
- 테스트 데이터로 실제 실행 → exit code + 출력 검증

상세: `references/criteria-script.md`

### 4.4 코드 검증

**정적 분석:**
- acceptance_criteria 충족 여부 (PROGRESS.json 참조)
- 가드레일 위반 없는가 (하드코딩 시크릿, scope 이탈)
- 타입 체크 통과 (언어별 도구: `criteria-code.md` §0 매핑 참조)

**동적 실행:**
- 테스트 실행 (언어별 러너: `criteria-code.md` §0 매핑 참조)
- lint 통과 (언어별 린터: `criteria-code.md` §0 매핑 참조)

상세: `references/criteria-code.md`

## 5. 검증 리포트

### PASS

```
✅ ralpi 검증 PASS

대상: {파일 경로}
타입: {PRD | SKILL | 스크립트 | 코드}
검증 횟수: {N}회
발견 이슈: {M}건 (모두 수정됨)

결과:
  - 정적 분석: ✓
  - 동적 실행: ✓ (해당 시)
```

### FAIL (7회 초과)

```
🔴 ralpi 검증 FAIL — 최대 시도 초과

대상: {파일 경로}
타입: {타입}
검증 횟수: 7회
미해결 이슈 ({K}건):
  1. {이슈 설명}
  2. {이슈 설명}

수동 수정이 필요합니다.
```

## 6. 결정표

| 상황 | 동작 |
|------|------|
| 아티팩트 경로 제공됨 | 타입 감지 → 검증 루프 |
| 타입 불명확 | 사용자에게 확인 |
| 정적 분석 이슈 발견 | 수정 → 재검증 |
| 동적 실행 실패 | 수정 → 재실행 |
| 7회 초과 | STOP + 미해결 이슈 리포트 |
| PASS | 리포트 출력 후 종료 |

## 7. 맥락 예산

| 시점 | 읽는 파일 | 읽지 않는 파일 |
|------|----------|-------------|
| **타입 감지** | 대상 파일 첫 50줄 | criteria-*.md (아직 불필요) |
| **검증 루프** | `criteria-{type}.md` 1개만 | 다른 타입의 criteria |
| **수정** | 대상 파일 전체 | criteria (이미 로드됨) |

**규칙**: criteria 파일은 **동시에 1개만**. 타입이 확정되면 해당 criteria만 로드.
