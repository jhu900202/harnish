# forki — 의사결정 강제

핵심 패턴: **2지선택 → 역할 분해 → trade-off → 강제 선택**.
이건 설명 스킬이 아니다. **의사결정 스킬**이다.

## 모드

**HITL 전용.** forki는 자율 모드가 없다. 결정자는 항상 사용자다.
스킬은 구조화하고, 제안하고, 도전한다 — 그러나 모든 게이트의 최종 답은 사용자의 것이다.
LLM은 후보를 제안할 수 있으나, **다음 단계로 진행하기 전 사용자의 확인이 반드시 필요하다.**

## Step 1: 2지선택 정의

선택을 **정확히 2개**로 진술한다. 3개 안 됨, "그리고 또..." 안 됨.

- 3개 이상 → 2개로 압축 (최선 vs 차선, 또는 "A 한다" vs "그 외 전부")
- "상황에 따라" → 결정 아님. **사용자에게 제약을 묻는다.**
- 대상이 모호 → **사용자에게 묻는다. 추측 금지.**

**HITL 프롬프트**:
> "선택을 다음과 같이 읽었습니다: **A) {옵션 A}** vs **B) {옵션 B}**. 맞으면 확인, 아니면 정정해주세요."

명시적 확인을 기다린다. 침묵이나 "응"으로 진행하지 않는다.

출력: `Option A vs Option B`. 한 줄.

## Step 2: 한 줄 환원

문제를 **한 문장**으로 압축한다. 다음 중 하나의 형태로:

- "누가 X를 **실행**하는가?"
- "누가 X를 **책임**지는가?"
- "X가 일어나면 **무엇이 변하는가**?"

**HITL 프롬프트**:
> "환원된 형태: *{한 문장}*. 이게 진짜 질문 맞나요?"

거부하면 → 대안 환원 제안. 또 거부되면 → Step 1로 (2지선택 자체가 잘못됨).

환원이 안 되면 → 문제 정의가 잘못됨. Step 1로 돌아간다.

## Step 3: 역할 분해 (D/E/V/R)

각 옵션에 대해 **4가지 역할** 모두 채운다:

| 역할 | 질문 |
|------|------|
| **Decision** | 누가 판단하는가 |
| **Execution** | 누가 실행하는가 |
| **Validation** | 누가 검증하는가 |
| **Recovery** | 누가 복구하는가 |

빈 칸 = 숨은 리스크. 이름을 강제한다 (사람, 시스템, 에이전트, "아무도 없음"). "아무도 없음"은 유효하고 중요한 답이다.

**HITL 프롬프트** (8칸 = 4역할 × 2옵션):
> "다음 표를 채워주세요. 이름, 역할, 시스템, 또는 '아무도 없음' 중 하나. 빈 칸은 허용되지 않습니다."
> ```
> | 역할        | Option A | Option B |
> |-------------|----------|----------|
> | Decision    | ?        | ?        |
> | Execution   | ?        | ?        |
> | Validation  | ?        | ?        |
> | Recovery    | ?        | ?        |
> ```

LLM은 초안을 **제안**할 수 있으나, 사용자가 모든 칸을 확인하거나 덮어써야 한다. 응답 후에도 `?`가 남으면 → 빈 칸만 다시 묻는다.

**이게 verdict gate다. Step 4와 7은 scaffold.**

## Step 4: 3가지 예시 (scaffold)

같은 구조를 **성격이 다른 3가지 구체 사례**에 적용한다.

목적: 구조가 진짜인지 우연 패턴인지 구분. 1개로는 패턴 매칭 착각 가능.

**HITL 프롬프트**:
> "이 D/E/V/R 구조를 공유하는 3가지 예시:
> 1. {사례 1} — {적용}
> 2. {사례 2} — {적용}
> 3. {사례 3} — {적용}
>
> 와닿나요? 다른 예시 찾을까요? (`skip`이라고 답하면 건너뜀)"

생략 가능: 문제가 너무 구체적이어서 예시가 정보를 추가하지 않을 때, 또는 사용자가 `skip`이라고 응답할 때. 생략 사실은 보고에 기록한다.

## Step 5: Trade-off 도출

각 옵션이 **얻는 것**과 **잃는 것**을 진술한다:

```
Option A: gains {X}, loses {Y}
Option B: gains {Y}, loses {X}
```

흔한 축: 유연성 ↔ 안정성, 탐색 ↔ 검증, 속도 ↔ 안전, 자율 ↔ 통제.

**HITL 프롬프트**:
> "Trade-off 초안:
> - A는 *{X}*를 얻고 *{Y}*를 잃음
> - B는 *{Y}*를 얻고 *{X}*를 잃음
>
> 이게 너에게 실제로 의미 있는 비용이 맞나요?"

Trade-off는 **사용자의 것**이지 LLM의 것이 아니다. "사실 X는 신경 안 써"라고 하면 → 그 축을 지우고 새 trade-off 제안 (Step 5 내에서, Step 3으로 돌아가지 않음).

두 옵션이 같은 걸 얻거나 같은 걸 잃으면 → 진짜 trade-off 아님. **Step 3으로 돌아간다** (역할 분해가 불완전).

## Step 6: 강제 선택

**하나** 고른다. 한 줄로:

> **선택**: Option {A | B}. 이유: {Step 3 또는 5에서 나온 구조적 이유 한 줄}.

**HITL 프롬프트** (이게 진짜 게이트):
> "표와 trade-off를 보고 **어느 쪽?** A 또는 B로 답하세요. '둘 다', '상황에 따라' 안 됩니다."

LLM은 사용자 대신 답할 수 **없다**. 구조적 경향("표상으로는 A가...")은 말할 수 있으나, 판정은 사용자의 것. "네가 골라줘"라고 하면 → 답: *"forki는 너 대신 결정할 수 없습니다. A 또는 B를 선택하세요."*

"둘 다", "상황에 따라", "나중에" 안 됨. 진짜 못 고르면 구조가 불완전한 것 — Step 3으로.

## Step 7: 이해 검증 (scaffold)

결정자에게 소크라테스식 3질문.

**HITL 프롬프트**:
> "빠른 확인 (`skip`으로 건너뛰기):
> 1. *{X}*는 무엇인가요? (정의)
> 2. A와 B의 진짜 차이는? (대비)
> 3. 각 옵션에서 누가 무엇을 하나요? (역할 회상)"

3개 다 못 답하면 → 결정이 **내재화 안 됨**. Step 3 다시 걸음.

생략 가능: 사용자가 `skip`이라고 응답할 때.

## 사용자 응답 해석 규칙

모든 HITL 프롬프트에 적용.

**명확한 응답 (진행)**:
- "A" / "B" / "Option A" → 적절한 위치에서 선택 또는 확정
- 표 셀 직접 수정 / 명시적 덮어쓰기 → 그대로 사용
- "응 근데 X를 Y로 바꿔" → 변경 적용 후 재확인

**모호한 응답 (재질문 필수)**:
- "응", "그래", "오케이", "ㅇㅇ", "Yeah" → **확정으로 해석 금지.** 해당 항목 다시 묻기.
- "둘 다", "어느 쪽이든", "상관없어" → Step 6에서 무효. 다시 진술: *"A 또는 B를 고르세요."*
- "상황에 따라" → 묻기: *"어떤 상황에 따라?"* — 그 제약이 Step 1 입력이 됨.
- "네가 골라줘" / "알아서 해줘" → **고르지 않음.** 답: *"forki는 너 대신 결정할 수 없습니다."*
- "몰라" / "I don't know" (Step 3 셀) → 묻기: *"답이 '아무도 없음'인가요, 아니면 더 생각해봐야 하나요?"* — `아무도 없음` 또는 이름이 나올 때만 진행.
- 침묵 / 무관한 응답 → 같은 프롬프트 1회 반복. 2번 무시되면 종료: *"forki 일시정지. {항목}에 대한 입장이 생기면 재개하세요."*

**탈출 응답 (종료)**:
- "그만" / "취소" → `forki aborted at Step {N}.`로 종료
- "다시" → Step 1로 복귀

## Context Budget

| 시점 | 읽는 것 |
|------|---------|
| Step 1 (2지선택) | 사용자 입력만. 파일 읽기 없음. |
| Step 2 (환원) | 이미 로드된 컨텍스트. |
| Step 3 (D/E/V/R) | 이미 로드된 컨텍스트. |
| Step 4 (예시) | 기억/알려진 사례. 사용자 요청 없으면 새 파일 읽기 없음. |
| Step 5–6 | 파일 읽기 없음. |
| Step 7 (검증) | 파일 읽기 없음. |

forki는 **사고** 스킬이지, 읽기 스킬이 아니다. 더 읽는다고 결정이 잘 되지 않는다.

## 금지

- Step 1에서 2개 초과 옵션
- Step 3 건너뛰기. D/E/V/R 표가 verdict gate.
- Step 4 또는 7을 게이트로 취급. 둘 다 scaffold이며 결정을 막을 수 없다.
- **LLM이 Step 6에서 최종 선택을 함.** 결정은 항상 사용자의 것.
- "응/오케이/ㅇㅇ"를 셀 채우기나 최종 선택으로 해석. 항상 해당 항목 재질문.
- 침묵 위에 다음 단계 진행
- "둘 다" / "상황에 따라" / "나중에" — 최종 선택으로 금지
- 결정 전에 "정보 더 모으겠다"며 파일 읽기 (사용자 명시 요청 없는 한)
- 두 옵션이 같은 걸 얻거나 같은 걸 잃는 trade-off
- "참고로..." 식 보충 정보
- 장황한 결정 보고
